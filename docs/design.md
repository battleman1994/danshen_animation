## Problem Statement

构建一个垂直品类的 AI 视频转化工具：用户输入文案/视频/链接，一键转化为以可爱动物（猫、熊、狐狸、熊猫等）为角色的动漫搞笑短视频，约 1 分钟，发布到抖音等平台获取社交反馈。现有通用 AI 视频工具（Runway、剪映等）功能泛化，不服务于"可爱小动物搞笑视频"这个细分场景。

## What Makes This Cool

- 输入一段搞笑台词，两只猫立刻演出来——像 UGC 但其实是 AI 生成的
- 把任意视频"动漫化"成动物角色重新演绎
- 最终交付的不是视频编辑工具，而是**内容工厂**——从文案到可发布的成品视频，一步到位
- 社交验证闭环：发到抖音 → 评论"哈哈哈太可爱了" → 多巴胺反馈

## Constraints

- 单人/小团队开发
- 核心瓶颈：视频生成质量（角色一致性、口型同步、动画流畅度）
- 已有 Rust/Dioxus 桌面端 + Python FastAPI 后端架构外壳
- 五阶段流水线模块（extractor/adapter/character/voice/composer）已搭建但未打通
- 视频生成流水线统一命名为：ScriptGen → CharacterGen → VoiceGen → Composer（对应现有 backend/src/pipeline/ 下的 adapter/character/voice/composer，extractor 暂不启用因为 V1 只做文字输入）

## Premises

1. **AGREED**: 通用 AI 视频工具不服务"动物角色搞笑视频"这个垂直场景，存在真实差距
2. **AGREED**: 视频生成流水线是核心瓶颈，其他一切（UI、多平台输入）在解决生成质量之前是次要的
3. **AGREED**: 先聚焦"文字→可爱动物动漫视频"一个场景做到极致
   - **执行约束**: B+C 并行中，Phase 1（视频生成流水线）必须先于 Phase 2（抖音小程序）交付 V1 质量。在流水线产出可转发视频之前，小程序端只做项目初始化，不投入主要精力。

## Cross-Model Perspective

独立子代理的建议（Claude 子代理，无上下文共享）：

- **最酷未考虑版本**: 直播实时动物头像覆盖层——观众看到主播的脸被替换为实时动漫动物表情
- **核心驱动力**: "别人觉得我的东西好可爱"——社交反馈是多巴胺来源
- **50%方案**: F5-TTS（情感语音合成）+ MimicMotion/AnimateAnyone（姿态驱动视频）提供基础能力；另外 50% 是角色一致性（SD LoRA 微调）+ 动漫渲染管线 + 脚本分镜分解
- **周末构建**: 放弃现有架构，Gradio + GPT-4o-mini 生成脚本 + ComfyUI/AnimateDiff 工作流 → ffmpeg 拼接

**Claude 的分析**: 同意周末验证方案的最小路径建议，但不同意放弃现有架构——毕竟已有可工作的前后端外壳。直播实时换脸作为长期愿景保留。

## Approaches Considered

### Approach A: Minimal MVP（已否决）
Gradio + ComfyUI + AnimateDiff，放弃现有架构，只做一条链路验证输出质量。
**否决原因**: 用户已有完整的 Rust/Dioxus + FastAPI 架构，放弃重建不划算。

### Approach B: Pipeline First — 保留现有架构，打通视频生成
保留 Rust/Dioxus 桌面端 + FastAPI 后端，集中全部精力打通 ComfyUI/AnimateDiff 视频生成流水线。后端输出标准 API，桌面端作为主要消费方。

### Approach C: Platform Wedge — 抖音小程序
先做抖音小程序前端，直接面向发布场景，获取真实用户反馈最快。后端提供视频生成服务。

## Recommended Approach: B + C 并行

**架构决策**: FastAPI 后端作为共享视频生成服务层，同时支撑两个前端：
- **桌面端**（现有 Rust/Dioxus）——开发调试主界面，功能完整
- **抖音小程序端**（新建）——面向最终用户，直接发布到平台

### 执行顺序

1. **Phase 1: 打通视频生成流水线（核心瓶颈）**
   - 用 ComfyUI + AnimateDiff 搭建基础角色动画管线
   - SD 1.5 + 动漫 LoRA 微调固定角色（狸花猫、棕熊、小狐狸）——选择 SD 1.5 作为起点，因为生态成熟、推理快、VRAM 需求低（8GB 可跑）。注：SDXL 生态（Pony/Illustrious/NoobAI）动漫质量更高，作为 V2 升级路径
   - **LoRA 数据策略**: 每个角色收集 30-50 张高质量动漫参考图（Danbooru/e621 标签筛选 + AI 辅助生成底图），使用 kohya-ss/sd-scripts 训练，约 3000-5000 steps，单角色训练耗时约 1-2 小时（4090 GPU）
   - **AnimateDiff 动物适配风险**: 开源 AnimateDiff 模型主要训练在人运动数据上，动物/动漫角色运动可能效果不佳。应对：先测试社区动漫 AnimateDiff 模型（CivitAI 动漫运动 checkpoint），如不理想，V1 降级为"静态角色 + 镜头运动（Ken Burns 效果）+ 字幕"替代全帧动画
   - **跨片段一致性策略**: 所有片段使用同一 SD seed + 同一 LoRA + 同一 prompt 模板描述角色外观。ComfyUI workflow 中锁定 character prompt 和 LoRA weight，仅变动动作/表情描述。片段间使用相同背景图
   - **LLM 分镜格式**: 输入 200 字台词脚本 → LLM 输出结构化分镜 JSON
     ```json
     {
       "scenes": [
         {
           "index": 1,
           "duration_s": 4,
           "dialogue": "今天天气真好呀！",
           "character": "tabby_cat",
           "emotion": "happy",
           "action": "waving_paw",
           "background": "sunny_park",
           "camera": "medium_shot"
         }
       ],
       "total_duration_s": 32,
       "style": "funny"
     }
     ```
     使用 GPT-4o-mini，prompt 模板固化在 adapter 模块中。分镜规则：每段 3-5s，总时长不超过 60s，角色不超过 2 个
   - **ComfyUI API 集成**: 后端通过 ComfyUI 的 REST API（默认端口 8188）驱动生成。Workflow JSON 模板预存在 backend/src/pipeline/character.py，后端按分镜动态替换 prompt/seed/LoRA 参数后 POST 到 `/api/prompt`，轮询 `/api/history/{id}` 获取输出图片。流程图：FastAPI → POST ComfyUI workflow → 轮询完成 → 下载 PNG → 进入下一片段
   - **V1 TTS 策略**：Edge-TTS 作为主力（零成本、中文稳定），F5-TTS 作为 V2 升级路径（情感控制更好但生产工具链薄）。跳过口型同步——Wav2Lip 对动漫面部效果未经验证，V1 阶段用字幕替代
   - FFmpeg 音视频合成 + 字幕叠加
   - 目标：输入一段 200 字搞笑台词 → 输出一个 30-60s 可看的动物动漫视频
   - **时间盒**: Phase 1 设 2 周死线。2 周后未达标 → 触发降级计划：静态角色 + Ken Burns 镜头运动 + Edge-TTS 语音 + 字幕，先交付可发布版本
   - **交付物**：3 个最佳输出视频，作为"这就是我们要的质量"的质量标杆

2. **Phase 2: 抖音小程序前端**（Phase 1 产出 V1 质量视频后启动）
   - 在 Phase 1 进行期间，仅初始化小程序项目框架，不写业务逻辑
   - Phase 1 V1 验收通过后，再投入小程序业务开发
   - Taro 或 uni-app 框架（一套代码多端）
   - 核心页面：首页输入 → 角色选择 → 生成进度 → 视频预览 → 一键发布
   - 调用后端 REST API（与桌面端共享）

3. **Phase 3: 桌面端优化**
   - 现有 Rust/Dioxus 前端接入新 API
   - 作为"开发者面板"——更完整的参数调节、调试预览

### 技术栈细化

| 层 | 技术 | 理由 |
|------|------|------|
| AI 角色生成 | ComfyUI + AnimateDiff + SD 1.5 LoRA | SD 1.5 动漫 LoRA 生态最丰富 |
| TTS | Edge-TTS（V1 主力）/ F5-TTS（V2 升级） | Edge-TTS 零成本，中文稳定；F5-TTS 情感控制好但生产工具链薄 |
| 口型同步 | V1 跳过，用字幕替代 | Wav2Lip 对动漫角色效果未验证，不阻塞 V1 交付 |
| 脚本生成 | GPT-4o-mini / DeepSeek | 低成本，中文脚本质量好 |
| 后端 | 现有 FastAPI + Celery + Redis | 已有，无需重造 |
| 桌面端 | 现有 Rust/Dioxus | 已有 |
| 小程序端 | Taro 3.x + React | 一套代码编译到抖音/微信 |

## Reviewer Concerns

以下问题在独立评审中被提出，属于已知风险，非阻塞：

1. **ComfyUI 错误处理不足**: 文档仅描述了 ComfyUI 集成的 happy path。生产环境中需要处理：POST 超时重试（指数退避）、GPU OOM 自动降分辨率、队列积压监控。建议在 Phase 1 后期加入。
2. **V1 质量评估**: "有人愿意转发"是主观标准。建议 Phase 1 验收时找 3 个人独立评分（角色一致性/动作流畅度/音频质量/搞笑程度，各 1-5 分），均分 ≥ 3.5 视为通过。
3. **SD 1.5 vs SDXL**: 文档选择 SD 1.5，但 Pony/Illustrious/NoobAI（基于 SDXL）的动漫 LoRA 生态质量更高。如 SD 1.5 角色一致性不理想，应升级到 SDXL。V1 用 SD 1.5 降低启动成本是合理的，但需在技术探针阶段快速验证。

## Success Criteria

- **V1 验收**: 输入一段 200 字搞笑台词 → 输出一个带字幕、有人愿意转发到群里的 30-60s 动物动漫视频
- **V2 验收**: 抖音小程序上线，第一个非自己发布的视频获得 100+ 点赞

## What I noticed about how you think

- "两只可爱的猫演绎"——你对产品画面有非常具体的想象，不是在描述功能而是在描述**场景**，这是做消费级产品最重要的直觉
- 你已经在架构层面做了正确的事——五阶段流水线、FastAPI、跨平台考虑——但在还没验证核心输出时，这些架构投资就是等待价值释放的资产
- 当我说"先聚焦文字转视频"，你同意了，但随后立刻选择 B+C 并行——你在战略上接受聚焦但在执行上想要多线推进。这个张力是好的，只要不被"同时做太多"拖慢第一次输出
- 你说"哈哈哈太可爱了"的时候，描述的不是你的产品，而是你期待的用户反应。这意味着你真正的北极星指标不是功能完整性，而是**一个陌生人看到视频后的第一反应**
